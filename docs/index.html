Preamble

Why an open standard

OpenClaw proved the pattern. But 430,000 lines of TypeScript in a single Node.js process isn't for everyone. The pattern itself — four phases of agent execution forming a persistent loop — is the real innovation. This specification extracts that pattern into a vendor-neutral protocol so anyone can build their own assistant in any language, with total control over their code, their data, and their security posture.

Design principles

Normative requirements

  • Language-agnostic. Every interface is defined as a behavioral contract (input shape → output shape), not a library API. Implement in Go, Rust, Python, TypeScript, Zig, C — whatever your threat model and deployment target demand.
  • Vendor-neutral. No primitive depends on a specific LLM provider, database, messaging platform, or cloud service. Every dependency is behind a pluggable interface.
  • File-first persistence. All durable state (memory, config, persona) MUST be representable as plain files on a filesystem. Databases are an optimization layer, not a requirement.
  • Auditable by default. Every agent action MUST produce a trace record. A human MUST be able to reconstruct what the agent did, why, and with what data — after the fact, without special tools.
  • Progressive trust. The system MUST start with zero permissions and require explicit grants for each capability. No tool, channel, or memory scope is enabled by default.

The LTDR message

All four phases operate on a single shared data structure: the LTDR Message. This is the normalized representation that flows through the entire system. Every channel adapter produces it, every agent consumes it, every tool result extends it.

Phase 1

Listen — channel adapters

The Listen phase normalizes input from the outside world into LTDR Messages. Every messaging platform, CLI, voice interface, or webhook becomes a channel adapter that speaks one common protocol.

Specification

A ChannelAdapter is any component that satisfies this behavioral contract:

  • start() — Begin listening for inbound messages. Establish platform-specific auth (OAuth, bot token, QR code). Return only after the connection is live.
  • stop() — Gracefully disconnect. Flush any pending outbound messages. Release platform resources.
  • on_message(callback) — Register a handler that receives normalized LTDRMessage objects. The adapter is responsible for translating platform-specific payloads into this shape.
  • send(message, target) — Deliver an outbound message to a specific user/group. The adapter translates from LTDR format back to platform-specific formatting (Markdown → Block Kit for Slack, etc.).
  • health() — Return connection status. Used by the session manager for circuit-breaking.

Security requirement: Adapters are untrusted input boundaries. All inbound content MUST be treated as potentially malicious. Sanitize text, validate attachment MIME types against actual content (not just headers), and enforce size limits before passing messages upstream.

Phase 2

Think — reasoning engine

The Think phase assembles context, invokes an LLM, and decides what to do next. It implements a reasoning loop (ReAct pattern) that alternates between thinking, acting, and observing until a final response is ready.

Specification

The Think phase consists of two subcomponents:

  • ContextAssembler — Composes the full prompt from system instructions, persona, memory, chat history, tool definitions, and the current message. Token-budget aware: MUST truncate or summarize when approaching the model's context limit.
  • ReasoningLoop — Sends the assembled context to a ModelProvider, receives a response, and decides: (a) if the response contains a tool call → dispatch to the Do phase, append result, loop back; (b) if the response is a final text → exit loop, return to Listen phase for delivery.

Security requirement: The model provider is a network boundary. All requests MUST be sent over TLS. API keys MUST NOT appear in logs, traces, or memory files. Responses from the model MUST be treated as untrusted — validate tool call schemas before execution.

Context assembler

Reasoning loop (ReAct)

// The ReAct loop — the core of every LTDR assistant. // Think → (Act → Observe)* → Respond type ModelProvider interface { // Send assembled context, get a response (text or tool calls) Complete(ctx context.Context, window *ContextWindow) (*ModelResponse, error) } type ModelResponse struct { Text string // Final response text (if no tool calls) ToolCalls []ToolCall // Requested tool invocations } type ToolCall struct { ID string // Unique call ID for matching results Name string // Tool name from registry Args json.RawMessage // Arguments as JSON } func RunReActLoop( ctx context.Context, assembler *ContextAssembler, provider ModelProvider, executor ToolExecutor, // The "Do" phase session *Session, inbound Message, maxTurns int, // Safety limit ) (string, error) { window, err := assembler.Assemble(session, inbound) if err != nil { return "", err } for turn := 0; turn < maxTurns; turn++ { // THINK: ask the model resp, err := provider.Complete(ctx, window) if err != nil { return "", err } // No tool calls → final response → exit loop if len(resp.ToolCalls) == 0 { return resp.Text, nil } // ACT + OBSERVE: execute each tool, append results for _, call := range resp.ToolCalls { result, err := executor.Execute(ctx, call) if err != nil { result = &ToolResult{Error: err.Error()} } // Append tool result to context for next iteration window.Messages = append(window.Messages, ChatMessage{Role: "assistant", ToolCalls: []ToolCall{call}}, ChatMessage{Role: "tool", ToolCallID: call.ID, Content: result.Output}, ) } } return "[max reasoning turns exceeded]", nil }

Phase 3

Do — tool execution

The Do phase executes actions in the real world: web searches, shell commands, API calls, file operations. Every tool runs through a sandboxed executor with capability-based permissions.

Specification

  • ToolRegistry — A map of tool name → tool definition (JSON Schema) + tool handler (the function that actually executes). Tools are registered at startup; the registry is immutable during a session.
  • ToolExecutor — Takes a ToolCall from the Think phase, validates the arguments against the registered schema, checks permissions, executes the handler in a sandbox, and returns a ToolResult.
  • Sandbox — Tool handlers MUST NOT run in the main process. Options: WASM module (IronClaw pattern), subprocess with resource limits, Docker container, or at minimum a restrictive syscall filter. The sandbox grants explicit capabilities: filesystem paths, network hosts, environment variables.

Security requirement: Default deny. A tool has zero capabilities until explicitly granted. Network access is restricted to approved hosts. Filesystem access is restricted to approved paths. Time limits prevent runaway execution. Every execution is logged with inputs, outputs, duration, and resource usage.

Phase 4

Remember — persistent memory

The Remember phase persists knowledge across sessions. Two layers: ephemeral daily notes (raw journal) and curated long-term memory (distilled knowledge). Hybrid search retrieves relevant context when needed.

Specification

The MemoryEngine satisfies this contract:

  • write_daily(user_id, content) — Append to today's daily note. Append-only; never overwrite.
  • write_curated(user_id, content) — Write to long-term memory. Idempotent updates by key/section.
  • load_session_context(user_id) — Return today's + yesterday's daily notes + full curated memory. Called once at session start.
  • search(user_id, query, opts) — Hybrid search across all memory files. Returns ranked results with source attribution.
  • flush_before_compaction(session) — The critical invariant. Before any context compaction, write all important context to daily notes. MUST be silent (no user-visible output).

File-first requirement: The canonical state MUST be readable plain files (Markdown recommended). Database indexes are acceleration layers, not sources of truth. A user MUST be able to cat their memory files and understand what the agent knows.

Privacy requirement: Curated memory (MEMORY.md) MUST only be loaded in private/DM sessions. It MUST never be injected into group chat context. Daily notes MAY be loaded in group contexts only if explicitly configured.

Swap implementations at compile time // or runtime via trait objects. #[async_trait] pub trait MemoryEngine: Send + Sync { async fn write_daily(&self, user_id: &str, content: &str) -> Result<(), MemError>; async fn write_curated(&self, user_id: &str, section: &str, content: &str) -> Result<(), MemError>; async fn load_session_context(&self, user_id: &str) -> Result<String, MemError>; async fn search(&self, user_id: &str, query: &str, opts: SearchOpts) -> Result<Vec<SearchResult>, MemError>; async fn flush_before_compaction(&self, session: &Session) -> Result<(), MemError>; } // File-based implementation uses: // - std::fs for file I/O (MEMORY.md, daily notes) // - rusqlite + sqlite-vec for vector search // - rusqlite FTS5 for keyword search // - chrono for temporal decay scoring pub struct FileMemory { workspace: PathBuf, index: SearchIndex, // SQLite-backed hybrid search } impl FileMemory { pub async fn write_daily(&self, user_id: &str, content: &str) -> Result<(), MemError> { let date = Utc::now().format("%Y-%m-%d").to_string(); let path = self.workspace.join("memory").join(format!("{date}.md")); let mut file = OpenOptions::new() .create(true).append(true) .mode(0o600) // Owner read/write only .open(&path)?; let time = Utc::now().format("%H:%M"); writeln!(file, "\n## {time}\n{content}")?; self.index.reindex_file(&path).await } }
# Memory engine — Protocol-based, file-first. # The search index (SQLite + embeddings) accelerates but never replaces files. class MemoryEngine(Protocol): async def write_daily(self, user_id: str, content: str) -> None: ... async def write_curated(self, user_id: str, section: str, content: str) -> None: ... async def load_session_context(self, user_id: str) -> str: ... async def search(self, user_id: str, query: str, opts: SearchOpts) -> list[SearchResult]: ... async def flush_before_compaction(self, session: Session) -> None: ... # ── File-based reference implementation ── class FileMemory: def __init__(self, workspace: Path, embedder: Embedder): self.workspace = workspace self.index = SearchIndex(workspace / ".index.sqlite", embedder) async def write_daily(self, user_id: str, content: str): date = datetime.now().strftime("%Y-%m-%d") path = self.workspace / "memory" / f"{date}.md" path.parent.mkdir(parents=True, exist_ok=True) time_str = datetime.now().strftime("%H:%M") with open(path, "a") as f: f.write(f"\n## {time_str}\n{content}\n") # Update search index await self.index.reindex_file(path) async def search(self, user_id, query, opts): # 1. Vector search (semantic similarity) vec_results = await self.index.vector_search(query, opts.max_results * 2) # 2. BM25 keyword search kw_results = await self.index.fts5_search(query, opts.max_results * 2) # 3. Hybrid merge with configurable weights merged = hybrid_rank(vec_results, kw_results, opts.vector_weight, opts.keyword_weight) # 4. Temporal decay (recent results rank higher) if opts.decay_half_life > 0: merged = apply_temporal_decay(merged, opts.decay_half_life) # 5. MMR diversity re-ranking (deduplicate similar results) return mmr_rerank(merged, opts.mmr_lambda, opts.max_results)
The compaction invariant: Before the context window is compressed, the agent MUST run a silent housekeeping turn to write any important context to daily notes. Without this, decisions and preferences that exist only in the conversation history will be lost when older messages are summarized. This is the single most important reliability mechanism in the entire standard.
Composition

Wiring it together

The four phases compose into a single event loop. Here's the complete orchestration — the main function that ties Listen, Think, Do, and Remember into a running assistant.

// main.go — A complete LTDR assistant in ~60 lines. // This IS the architecture. Everything else is implementation detail. func main() { cfg := LoadConfig("config.toml") ctx, cancel := signal.NotifyContext(context.Background(), os.Interrupt) defer cancel() // ── Initialize the four phases ── // REMEMBER: memory engine (file-based + SQLite index) memory := NewFileMemory(cfg.Workspace, NewEmbedder(cfg.EmbeddingProvider)) // DO: tool registry with sandboxed execution tools := NewToolRegistry() tools.Register(NewWebSearchTool(cfg.SearchAPIKey)) tools.Register(NewShellTool(cfg.ShellPermissions)) executor := NewSandboxedExecutor(tools) // THINK: context assembler + model provider persona := LoadFile(cfg.Workspace + "/SOUL.md") assembler := NewContextAssembler(cfg.MaxTokens, persona, tools.Definitions(), memory) provider := NewModelProvider(cfg.Model) // Claude, GPT, Ollama, LM Studio... // LISTEN: channel adapters sessions := NewSessionManager() adapters := []ChannelAdapter{ NewTelegramAdapter(cfg.TelegramToken), NewCLIAdapter(), // Terminal interface } // ── The event loop ── for _, adapter := range adapters { adapter.OnMessage(func(msg Message) { session := sessions.GetOrCreate(msg.Source.UserID) // Acquire the session lane (single-writer concurrency) session.Lane.Enqueue(func() { // THINK + DO: ReAct loop (calls tools as needed) response, err := RunReActLoop(ctx, assembler, provider, executor, session, msg, 10) if err != nil { response = "Sorry, I encountered an error: " + err.Error() } // REMEMBER: write conversation to daily notes memory.WriteDaily(msg.Source.UserID, fmt.Sprintf("User: %s\nAssistant: %s", msg.Content.Text, response)) // LISTEN (outbound): send response back through the same channel adapter.Send(ctx, Message{Content: Content{Text: response}}, msg.Source.UserID) }) }) adapter.Start(ctx) } // Block until shutdown signal <-ctx.Done() for _, a := range adapters { a.Stop(ctx) } }

// main.rs — A complete LTDR assistant. // Compile-time safety for the entire pipeline. #[tokio::main] async fn main() -> Result<(), Box<dyn std::error::Error>> { let cfg = Config::load("config.toml")?; // REMEMBER let memory = FileMemory::new(&cfg.workspace, Embedder::new(&cfg.embedding)); // DO let mut registry = ToolRegistry::new(); registry.register(WebSearchTool::new(&cfg.search_key)); registry.register(ShellTool::new(&cfg.shell_perms)); let executor = SandboxedExecutor::new(registry); // THINK let persona = std::fs::read_to_string(cfg.workspace.join("SOUL.md"))?; let assembler = ContextAssembler::new(cfg.max_tokens, persona, &executor); let provider = ModelProvider::new(&cfg.model); // LISTEN let sessions = Arc::new(SessionManager::new()); let (tx, mut rx) = tokio::sync::mpsc::channel::<(Message, Arc<dyn ChannelAdapter>)>(256); // Start adapters let tg = Arc::new(TelegramAdapter::new(&cfg.telegram_token, tx.clone())); tg.start().await?; // Event loop while let Some((msg, adapter)) = rx.recv().await { let session = sessions.get_or_create(&msg.source.user_id); let _guard = session.lane.lock().await; // THINK + DO let response = react_loop(&assembler, &provider, &executor, &session, &msg, 10).await?; // REMEMBER memory.write_daily(&msg.source.user_id, &format!("User: {}\nAssistant: {}", msg.content.text, response)).await?; // LISTEN (outbound) adapter.send(&Message::text(&response), &msg.source.user_id).await?; } Ok(()) }
That's the entire architecture. Everything else — which embedding model, which LLM provider, which messaging platform, which database, which sandbox technology — is an implementation choice behind a pluggable interface. The LTDR standard defines the contracts between phases, not the implementations within them. Build with the tools your threat model demands.

Example implementations

Four reference architectures showing how the LTDR standard adapts to different deployment targets, security requirements, and team sizes.

Minimal: CLI-only assistant

~200 lines. Single Go binary. No messaging platform — just stdin/stdout. SQLite for memory. Ollama for local inference. Perfect learning project to understand the four phases before adding complexity.

// Entire stack: Listen: stdin readline loop Think: Ollama HTTP API Do: shell exec (sandboxed) Remember: flat Markdown files Search: grep (no vector DB) Binary: ~8MB, zero dependencies

File system layout (standard)

Every LTDR implementation uses this workspace structure. The files ARE the state — databases are indexes that can be rebuilt from the files.

~/.assistant/ ├── config.toml # All settings: model, channels, tools, search ├── SOUL.md # Agent persona and behavioral rules ├── MEMORY.md # Curated long-term knowledge (private sessions only) ├── memory/ │ ├── 2026-03-18.md # Today's daily notes (append-only) │ ├── 2026-03-17.md # Yesterday │ └── ... ├── sessions/ # Saved conversation transcripts (optional) ├── tools/ # Custom tool definitions + handlers ├── .index.sqlite # Search index (rebuild from files if lost) └── .audit/ # Trace logs for every agent action └── 2026-03-18.jsonl # One JSON line per action
Why file-first matters for security

When your agent's knowledge lives in plain Markdown files, you get properties that no database provides: you can diff what changed between sessions, grep for sensitive content before sharing, git version your agent's knowledge, rsync backups to encrypted storage, and — critically — read everything your agent knows with nothing but a text editor. There's no query language to learn, no admin console to compromise, no proprietary format to decode. If the agent "forgets" something, you open the file and see exactly why: either the write never happened, or the search index missed it. Transparent, auditable, debuggable.

Implementing progressive trust

Start every capability at level 0 (ask before every invocation). As you build confidence, promote to level 1 (ask for new action types, auto-approve known patterns), then level 2 (auto-approve within a policy boundary), then level 3 (fully autonomous within sandbox). Each promotion is an explicit edit to config.toml — not something the agent can self-escalate. The audit log records every action at every trust level, so you can retroactively review what autonomous decisions the agent made and demote if needed.

Agent scenarios

Where custom LTDR wins

OpenClaw is 430,000 lines of TypeScript you didn't write and can't fully audit. For many use cases, that's fine. For these scenarios, owning every line of code isn't a preference — it's a requirement. Each scenario maps the LTDR phases to domain-specific implementations that a general-purpose framework can't safely provide.

When to build your own: If your agent handles regulated data, runs air-gapped, requires domain-specific tool sandboxing, needs a custom memory topology, or must pass a compliance audit where "we use an open-source framework" isn't an acceptable answer — build with LTDR primitives. If you're automating personal tasks on a Mac Mini, OpenClaw is great.

Healthcare compliance agent

A clinical practice assistant that handles patient scheduling, triages symptom intake forms, drafts referral letters, and retrieves relevant treatment protocols. Handles PHI (Protected Health Information) under HIPAA — every data path must be auditable, encrypted, and access-controlled.

Why LTDR over a framework

HIPAA requires a Business Associate Agreement with every vendor that touches PHI. An open-source framework with 3,000+ community plugins (12% infection rate per CVE-2026-25253) cannot satisfy this. You need to audit every line that touches patient data. A custom LTDR implementation in ~2,000 lines of Rust is auditable in a day.

Listen

Secure patient portal (HTTPS-only web form) + encrypted SMS via Twilio HIPAA tier. No consumer messaging platforms — WhatsApp, Telegram, and Discord are explicitly excluded because they don't sign BAAs. Voice intake via on-premise speech-to-text (Whisper running locally, never cloud-transcribed).

Think

Model provider routed through a HIPAA-compliant API endpoint (Azure OpenAI with BAA, or on-premise inference). Context assembler strips all PHI identifiers before sending to cloud models — replace names with tokens, reconstruct on response. Memory injection limited to the current patient's record.

Do

Tools: EHR system queries (Epic/Cerner FHIR API), scheduling (calendar write with conflict detection), referral letter generation (template-fill, never freeform PHI in tool output), insurance eligibility check. Every tool call logged with patient ID, timestamp, and data fields accessed.

Remember

Memory encrypted at rest (AES-256). Per-patient memory isolation — no cross-patient search. Daily notes stored in HIPAA-compliant storage with automatic retention policies (7 years). Curated memory holds care protocols and provider preferences, never patient data directly.

Quantitative trading assistant

A market analysis agent that monitors real-time price feeds, evaluates options strategies against your portfolio Greeks, executes alerting rules, and drafts trade rationale memos. Needs sub-second data ingestion and must never autonomously execute trades without explicit human confirmation.

Why LTDR over a framework

Financial data feeds require persistent WebSocket connections with specific exchange protocols (FIX, binary). The Listen phase needs a custom adapter that speaks these protocols natively — not a consumer chat adapter. The Do phase must enforce a hard boundary: analysis tools run autonomously, but execution tools require interactive confirmation. A framework's uniform permission model can't express this split.

Listen

Three adapters: (1) Market data WebSocket (exchange-native binary protocol, not HTTP). (2) CLI terminal for direct commands. (3) Telegram for mobile alerts. The market data adapter normalizes ticks into LTDR Messages with metadata carrying the full order book snapshot.

Think

Local model (MLX on Apple Silicon) for real-time pattern recognition — cloud latency is unacceptable for market data. Cloud model (Claude) for complex multi-leg options analysis when latency isn't critical. Context includes current portfolio positions, Greeks, and active alerts from memory.

Do

Two trust tiers. Tier 1 (autonomous): price alerts, volatility calculations, P&L snapshots, technical indicator computation. Tier 2 (confirmation required): order placement, position modification, fund transfer. Tier 2 tools send a confirmation message to Telegram and block until the user responds.

Remember

Daily notes capture every trade rationale, market observation, and strategy adjustment — a built-in trading journal. Curated memory holds your risk parameters, position sizing rules, and historical strategy performance. Search surfaces past rationale: "Why did I exit that SPY put spread in January?"

SRE on-call incident responder

An operations agent that receives PagerDuty alerts, triages severity, executes diagnostic runbooks, correlates symptoms across services, and drafts incident postmortems. Runs in your infrastructure VPC — never routes production telemetry through external services.

Why LTDR over a framework

Incident response tools need direct access to internal APIs (Kubernetes, Prometheus, CloudWatch) that sit behind VPC boundaries. The agent must run inside your network perimeter — not on a developer's laptop connecting outward. A custom LTDR deployment as a container in your infrastructure cluster satisfies this. Additionally, shell execution during an incident must be tightly scoped: read-only diagnostic commands, never mutating operations, unless the operator explicitly escalates.

Listen

Webhook adapter receiving PagerDuty/OpsGenie alerts (JSON payload → LTDR Message). Slack adapter for interactive triage in the incident channel. CLI adapter on a bastion host for direct operator commands. Each adapter tags messages with severity level and affected service.

Think

Context assembler injects the on-call runbook for the affected service, recent deployment history (from memory), and current service dependency graph. Model reasons about which diagnostic steps to run and in what order. Chain-of-thought is logged for the postmortem.

Do

Tools: kubectl get pods (read-only), Prometheus range queries, CloudWatch log tailing, deployment history lookup, Slack message posting. All tools run with read-only Kubernetes RBAC. Mutating operations (kubectl rollout restart) require operator confirmation via Slack reaction.

Remember

Daily notes become the incident timeline automatically. Curated memory holds: service ownership map, known failure modes per service, past incident patterns ("Redis connection exhaustion happened 3 times in Q1 — always check max-connections first"). Postmortem drafts pull directly from the daily notes.

Permaculture & homestead manager

An offline-capable agent for managing a permaculture garden, seed library, harvest tracking, weather integration, and trade/barter inventory. Runs on a Raspberry Pi in a greenhouse with intermittent connectivity — must function without internet access for days at a time.

Why LTDR over a framework

This agent must work offline. OpenClaw requires a cloud LLM API for its agent runtime — no internet, no assistant. A custom LTDR implementation with a local model (Ollama on a Pi 5, or a quantized model on Apple Silicon) keeps the Think phase running regardless of connectivity. The Remember phase uses local SQLite exclusively — no cloud embedding API needed if you run sentence-transformers locally. When connectivity returns, a sync adapter pushes daily notes to a git remote.

Listen

CLI adapter on the Pi terminal. Local web UI (served via LAN). Optional: Telegram adapter that queues messages when offline and syncs when connectivity returns. IoT sensor adapter reading soil moisture, temperature, and light levels from GPIO/I2C sensors — each reading becomes an LTDR Message.

Think

Local model only — Qwen3 30B-A3B via Ollama (MoE for fast prefill on limited hardware) or a smaller 7B model on Pi 5. Context includes current season, active plantings from memory, recent sensor readings, and the planting calendar. No cloud dependency.

Do

Tools: Weather API (cached, with graceful offline fallback to last-known forecast). Seed database queries (local SQLite: variety, days-to-maturity, companion plants, trade value). Harvest logger (append to CSV). Irrigation controller (GPIO relay). Planting calendar generator (frost dates + succession intervals).

Remember

Daily notes capture: plantings, harvests, pest observations, weather notes, barter transactions. Curated memory holds: perennial locations, soil amendment history, seed saving schedules, trade partner preferences. Multi-year memory enables pattern recognition: "Tomato blight appeared in the same bed three years running — rotate to legumes."

Legal practice assistant

A law firm agent that manages case timelines, reviews document discovery, drafts correspondence, tracks billable hours, and retrieves relevant precedent. Client communications are protected by attorney-client privilege — the agent must enforce strict data isolation between matters.

Why LTDR over a framework

Attorney-client privilege requires that no client's data is ever accessible in another client's context. OpenClaw's memory search operates across all files in the workspace — a single workspace per firm means one client's privileged information could surface in another client's session. A custom LTDR implementation partitions memory by matter ID: each client/matter gets its own workspace directory, its own search index, and its own session scope. Cross-matter search is architecturally impossible, not just policy-blocked.

Listen

Encrypted email adapter (IMAP/S with per-matter routing rules). Secure client portal (web form with matter ID authentication). Internal Slack for attorney collaboration. Every inbound message is tagged with a matter ID that determines which workspace and session it routes to.

Think

Context assembler loads ONLY the current matter's memory — workspace isolation enforced at the filesystem level. Model provider uses a firm-hosted API endpoint (no client data to consumer cloud). Persona includes jurisdiction-specific rules and the firm's style guide for correspondence.

Do

Tools: Case management system API (Clio, PracticePanther). Document comparison (diff two contract versions). Precedent search (local vector index of firm's case library). Billable time logger (auto-tracks agent interaction time per matter). Calendar/deadline management with statute of limitations alerts.

Remember

Per-matter workspace: ~/.firm/matters/{matter_id}/ — each with its own MEMORY.md, daily notes, and search index. Firm-wide memory (non-privileged) holds: court procedures, filing requirements, judge preferences. Retention policies auto-archive closed matters after the applicable retention period.

Investigative research & OSINT agent

A research agent that monitors public data sources, correlates entities across datasets, tracks story leads, and manages source relationships. Source protection is paramount — the agent must never leak source identities across investigations or to external services.

Why LTDR over a framework

Source protection requires that the agent's network traffic is forensically clean — no DNS leaks, no unencrypted metadata, no third-party analytics. OpenClaw's skill ecosystem has had supply chain attacks (CVE-2026-25253) that exfiltrated data to unknown endpoints. A custom LTDR implementation routes ALL network traffic through Tor or a journalist-controlled VPN, with a deny-by-default firewall that blocks every host not explicitly approved. The Listen phase never touches consumer platforms that log metadata.

Listen

Signal adapter (end-to-end encrypted, metadata-resistant) for source communication. Secure drop web form (onion service) for anonymous tips. CLI on an air-gapped workstation for analysis sessions. No Telegram, no Discord, no Slack — metadata exposure is unacceptable.

Think

Local model exclusively — no cloud API calls during active investigation (prevents subpoena-discoverable server logs at a third party). Context includes the investigation timeline, entity relationship map from memory, and known public records. Model runs on local hardware with full-disk encryption.

Do

Tools: Public records search (PACER, SEC EDGAR, corporate registries) — all via Tor. Shodan/Censys queries for infrastructure correlation. Entity extraction (local NER model, never cloud). Link analysis visualization (local graph renderer). All tools route through the controlled network proxy.

Remember

Per-investigation workspace with encrypted filesystem (LUKS or FileVault). Source identities stored only in a separate, manually-accessed vault — never in the agent's searchable memory. Daily notes capture leads, dead ends, and verification status. Entity graph stored as structured data for cross-reference queries.

Government cloud infrastructure agent

An infrastructure-as-code assistant operating within a government cloud environment (FedRAMP, CMMC, IL4/IL5). Manages Terraform plans, reviews security configurations, generates compliance documentation, and monitors drift across multi-account landing zones.

Why LTDR over a framework

Government cloud environments have strict network egress controls — no arbitrary outbound connections. OpenClaw's plugin ecosystem, auto-update mechanism, and npm dependency chain would fail compliance review immediately. A custom LTDR implementation compiles to a single auditable binary (Go or Rust), runs inside the VPC with no internet egress except to the approved LLM API endpoint, and produces the audit trail required by CMMC Level 2 controls (AU-2, AU-3, AU-6, AU-12).

Listen

Microsoft Teams adapter (the standard in government environments) via approved Bot Framework registration. CLI adapter on authorized administrator workstations. Webhook adapter for CI/CD pipeline triggers (GitLab/GitHub Enterprise). No consumer messaging platforms permitted in the authorization boundary.

Think

Model provider routed through the cloud provider's FedRAMP-authorized AI endpoint (Azure Government OpenAI, or an on-premise model in the enclave). Context includes the current Terraform state, active security findings, and the compliance control matrix from memory. No data leaves the authorization boundary.

Do

Tools: terraform plan (read-only analysis, never auto-apply). Security Hub finding aggregation. Config rule evaluation against CMMC/NIST 800-171 controls. IAM policy analyzer (detect overly permissive roles). Compliance report generator (Markdown → PDF for auditors). All tools scoped to read-only IAM roles; write operations require human approval via Teams.

Remember

Daily notes capture every infrastructure change, security finding, and remediation action — the continuous monitoring artifact auditors need. Curated memory holds: account topology, service ownership, approved architectural patterns, exception justifications. Memory stored in an encrypted database with row-level access control — different teams see only their accounts.

Adaptive learning tutor

A per-student tutoring agent that adapts to learning pace, identifies knowledge gaps, generates practice problems, and reports progress to instructors. Handles student data under FERPA (Family Educational Rights and Privacy Act) — data isolation between students is legally required.

Why LTDR over a framework

FERPA requires that student educational records are not disclosed without consent. A shared-workspace agent where one student's struggle with quadratic equations could surface in another student's context is a FERPA violation. The LTDR approach: per-student workspace with isolated memory, shared curriculum content (read-only, not per-student), and instructor-scoped aggregate views that strip PII before reporting.

Listen

School LMS integration (Canvas, Moodle) — student messages arrive tagged with student ID and course. Web chat widget embedded in the course page (authenticated via school SSO). No consumer messaging — students interact through the school's authorized platforms only.

Think

Context assembler loads the student's current knowledge model from memory (which concepts they've mastered, which they're struggling with), the relevant curriculum unit, and their learning preference profile. Model adjusts explanation complexity based on demonstrated understanding — concrete examples for visual learners, formal proofs for abstract thinkers.

Do

Tools: Problem generator (parameterized templates per curriculum standard). Solution checker (symbolic math engine, not LLM — deterministic correctness). Hint scaffolder (progressive disclosure: nudge → concept → worked example). Progress reporter (aggregate stats to instructor, PII stripped). Spaced repetition scheduler.

Remember

Per-student workspace: /students/{student_id}/. Daily notes track: problems attempted, time-to-solution, hint usage, misconception patterns. Curated memory holds the evolving knowledge model: "Mastered: linear equations, slope-intercept form. Struggling: systems of equations — consistently confuses substitution with elimination." Instructor view aggregates across students without exposing individual session data.
